# 项目说明

## 一句话介绍

我构建了一个面向中国 A 股年报的大规模自动化因子研究流水线。它从沪深 300 年度成分股出发，批量下载和解析上市公司年报，自动抽取 CAPE 与 FCF Yield 所需的关键财务字段，经过异常修复、策略回填和人工兜底后，生成可直接用于量化研究的结构化数据，并进一步完成资产配置回测、参数实验和 live 区间评估。目前项目内已沉淀 **7,768 份结构化年报 JSON 结果**，并形成了从原始 PDF/MD 文档到策略验证的闭环。

## 1. 项目解决的核心痛点

### 1.1 非结构化年报数据很难直接用于量化研究

中国上市公司年报大多以 PDF 形式披露，且存在以下典型问题：

- 报表格式高度不统一，不同年份、不同公司、不同交易所口径差异很大。
- 关键字段分散在多个表中，且经常出现同义字段、跨页表格、单位切换、附注干扰。
- 早期年报里扫描件、OCR 质量差、表格断裂、图片化页面等问题非常常见。
- 单个字段不是“找到数字”就结束，还需要判断字段口径是否正确，例如：
  - `归属于母公司股东的净利润` 不能误提成 `利润总额`。
  - `购建固定资产、无形资产和其他长期资产支付的现金` 必须来自现金流量表主表，而不是附注或投资说明。

### 1.2 人工抽取的成本和错误率都不可接受

这个项目处理的不是几十份年报，而是覆盖多年样本的大批量历史报告。研究口径要求连续年份、跨公司一致性和可追溯性，如果完全依赖人工：

- 工作量极大，无法支撑多轮策略假设验证。
- 容易出现字段口径漂移，后续回测结果不稳定。
- 一旦策略定义改变，就必须重新回头核验大量历史文档。

### 1.3 传统“抽完数据就结束”的流程无法支撑研究闭环

真实的量化研究问题不是“把数据抽出来”这么简单，而是需要形成完整闭环：

1. 建样本。
2. 抓年报。
3. 提字段。
4. 修异常。
5. 合并与补齐。
6. 计算因子。
7. 生成仓位。
8. 回测并验证假设。

如果中间任何一步不能自动化或标准化，整个研究链条就会频繁断裂。

## 2. 项目产出与价值

这个项目最终不是一个单点抽取脚本，而是一套可复用的研究基础设施：

- 面向沪深 300 多年份样本，自动构建年度成分股和报告清单。
- 批量处理 PDF/Markdown 年报，提取 CAPE 与 FCF Yield 所需字段。
- 提供失败重试、人工修复归并、公司均值回填、缺失剔除等数据治理机制。
- 生成按年份组织的财报 CSV、配置比例 CSV、回测结果和分析报告。
- 在策略层已经完成：
  - 原始双因子仓位映射实验。
  - FCF 阈值搜索。
  - CAPE/FCF 对下一年收益率的拟合分析。
  - CAPE 对下一年最大回撤的建模分析。
  - 基于预测回撤的仓位实验。
  - 2011-2024 live 区间单独评估。

从项目定位上看，它解决的是“把大规模、低结构化、噪声极高的中国年报，变成可回测、可复验、可迭代的量化研究数据资产”。

## 3. 核心逻辑流

### 3.1 样本构建

项目首先按年份建立沪深 300 成分股样本，并保留每个年度的样本清单、上市年限筛选结果和后续下载清单。

### 3.2 年报获取与预处理

针对每年的样本公司，批量获取年报 PDF，并按需要转换为：

- 原始 PDF
- Markdown 文本
- 局部高精度图片片段

之所以保留多种中间表示，是因为不同模型对不同载体的鲁棒性不同：

- 纯 PDF 提取适合保留页面结构。
- Markdown 便于长文本检索、分块和上下文裁剪。
- 高精度图片适合处理 OCR 失真、表格断裂和复杂版式。

### 3.3 财务字段抽取

项目围绕以下核心字段进行抽取：

- 归母净利润
- 期末总股本
- 经营活动现金流量净额
- 资本性支出（CapEx）

并在字段层做口径约束，例如：

- 强制优先主表口径。
- 区分“期末股本”和“加权平均股本”。
- 区分“经营现金流主表值”和补充资料值。
- 对 CapEx 做现金流量表主表定位。

### 3.4 失败修复与数据治理

项目不是单次抽取，而是带恢复机制的长流程系统。对于失败样本，项目会继续走后续路径：

- 自动重跑
- 局部字段修补
- Markdown 辅助定位后重新识别
- PDF 局部图片增强后重识别
- 人工兜底
- 结果再归并
- 按公司历史均值规则回填可接受缺失
- 对无法修复的极少样本做剔除

这一步是项目真正落地的关键，因为真实世界年报数据质量远比“demo 级抽取”复杂。

### 3.5 因子计算与配置生成

在字段层结构化之后，项目继续向下游推进：

1. 结合 CPI 做盈利通胀调整。
2. 计算个股层 10 年平均真实盈利。
3. 聚合到指数层 CAPE 与 FCF Yield。
4. 根据国债收益率等参数生成年度权益配置比例。

### 3.6 策略回测与假设验证

最后项目会把因子研究直接接到策略层：

- 原始双因子线性/非线性仓位映射回测
- 与 100% 权益、月定投、永久组合等基准对比
- 搜索 FCF 阈值和仓位映射边界
- 分析因子对下一年收益与下一年最大回撤的解释力
- 构建“预测下一年最大回撤 -> 决定当年权益仓位”的新策略链路

也就是说，这不是“做完抽取就结束”的项目，而是完整覆盖了“数据获取 -> 清洗 -> 因子 -> 策略 -> 验证”的全链条。

## 4. 是否包含长链推理、多 Agent 协作

### 4.1 长链推理：包含，而且是项目核心

这个项目明显包含长链推理，但它不是体现在单条 prompt 很长，而是体现在**多阶段依赖的任务链**：

1. 从 PDF/MD 中定位正确财务表。
2. 在多个候选数字里识别符合目标口径的字段。
3. 判断单位、年度列和合并/母公司报表口径。
4. 对失败样本切换处理策略。
5. 将字段结果合并为年度结构化数据。
6. 再向下游完成因子计算与回测验证。

尤其在年报抽取环节，模型并不是“读文档然后随便回答”，而是需要持续执行：

- 检索
- 对齐
- 候选排除
- 口径判断
- 证据定位
- 结果标准化

因此它属于比较典型的**长链文档理解 + 结构化抽取 + 下游验证**场景。

### 4.2 多 Agent 协作：不是当前主架构核心

当前项目**不以多 Agent 协作作为核心架构**。它更接近：

- 单主控流程
- 多批处理脚本
- 多模型路线切换
- 监控脚本观察进度
- 人工兜底处理异常

换句话说，这个项目的重点不在“多个 Agent 相互辩论和分工”，而在于**把一个高噪声、长链路、容易中断的年报研究流程工程化**。

如果要归纳当前协作方式，更准确的表述是：

- **不是多 Agent 系统**
- **是单主控 + 多模型 + 多阶段恢复机制 + Human-in-the-loop 的研究流水线**

## 5. 当前工程特征

从仓库结构看，这个项目已经具备比较完整的工程化特征：

- 有分步骤脚本，例如 `step5_fetch_close_prices.py`、`step6_extract_financials.py`、`step7_compute_allocation.py`、`step8_backtest.py`
- 有多条抽取路线，例如：
  - `step6_extract_financials_qwen_pdf.py`
  - `step6_extract_financials_from_markdown.py`
  - `repair_gemma_pdf_hybrid_fields.py`
  - `repair_gemma_markdown_financials.py`
- 有监控与恢复脚本，例如：
  - `monitor_qwen_progress.py`
  - `monitor_gemma_markdown_progress.py`
  - `monitor_gemma_pdf_hybrid_progress.py`
- 有策略研究脚本，例如：
  - `run_allocation_transform_experiments.py`
  - `analyze_factor_return_models.py`
  - `analyze_factor_drawdown_models.py`
  - `run_drawdown_driven_allocation_experiments.py`
  - `evaluate_live_period_strategies.py`

这说明项目已经从“单次脚本实验”演化成一套持续可迭代的研究工程。
